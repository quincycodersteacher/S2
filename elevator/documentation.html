Elevator Simulation Project Documentation

This documentation provides a comprehensive summary of the classes and methods in the elevator simulation project. It is designed to allow a student to understand the functionality and recreate the project independently. The project simulates an elevator system in a building, including people moving between floors, elevator logic, and visual representations on both a canvas and the browser console.

Project Structure

The project consists of five files across two directories:

File Purpose
elevator.html Main HTML page. Sets up the UI (canvas, buttons, stat displays) and bootstraps the simulation.
js/queue.js Queue data structures: base Queue, FloorQueue, and ElevatorQueue.
js/elevator.js Core simulation classes: Person, Elevator, RegularElevator, Floor, Building, and Simulation.
js/elevatorcanvas.js Canvas renderer (ElevatorCanvas). Draws the building, shafts, elevators, people, and buttons on an HTML canvas.
js/elevatorconsole.js Console renderer (ElevatorConsole). Prints a text-based representation of the simulation to the browser console.

Step-by-Step Class Implementation

To recreate the project, implement the classes in the following order based on their dependencies. Each class includes an explanation of why it should be built at that step and what comes next.

1. Person Class

File: js/elevator.js

Why build this first: The Person class has no dependencies and represents the fundamental entity (people) that will be managed throughout the simulation. It's a simple data holder that other classes will reference.

Next: After Person, build the Queue class, as queues are needed to manage collections of people.

Represents an individual person in the simulation.

Properties:

  • name: A string identifier for the person (e.g., "a", "b").
  • originFloor: The floor number where the person starts.
  • destinationFloor: The floor number where the person wants to go.

Methods:

  • constructor(name, originFloor, destinationFloor): Initializes a new Person with the given name, origin floor, and destination floor.
  • equals(otherPerson): Checks if this person is equal to another based on name. Returns a boolean.
Testing Ideas
  • Test constructor: let p = new Person('a', 0, 2); console.assert(p.name === 'a' && p.originFloor === 0 && p.destinationFloor === 2);
  • Test equals: let p1 = new Person('a', 0, 1); let p2 = new Person('a', 1, 2); console.assert(p1.equals(p2) === true); let p3 = new Person('b', 0, 1); console.assert(p1.equals(p3) === false);

2. Queue Class

File: js/queue.js

Why build this next: Queues are fundamental data structures used to manage ordered collections of people in floors and elevators. Build this after Person since it will hold Person instances.

Next: Build FloorQueue, which extends Queue for floor-specific logic.

A basic queue data structure for managing people.

Properties:

  • people: An array holding the queued elements.

Methods:

  • constructor(): Initializes an empty queue with this.people = [].
  • enqueue(element): Adds an element to the end of the queue using Array.push().
  • dequeue(): Removes and returns the first element using Array.shift(), or null if empty.
  • peek(): Returns the first element without removing it, or null if empty.
  • isEmpty(): Returns true if the queue has no elements.
  • size(): Returns the number of elements in the queue.
Testing Ideas
  • Test constructor and isEmpty: let q = new Queue(); console.assert(q.isEmpty() === true && q.size() === 0);
  • Test enqueue and dequeue: q.enqueue('a'); q.enqueue('b'); console.assert(q.size() === 2 && q.dequeue() === 'a' && q.size() === 1);
  • Test peek: console.assert(q.peek() === 'b' && q.size() === 1);

3. FloorQueue Class

File: js/queue.js

Why build this next: FloorQueue extends Queue to provide floor-specific dequeue methods for handling people based on their destination relative to the floor. It depends on Queue.

Next: Build ElevatorQueue, which also extends Queue for elevator-specific logic.

A specialized queue for floors, extending Queue.

Methods:

  • Inherits constructor from Queue.
  • dequeueNextWithDestinationAbove(floor): Iterates through the queue and removes and returns the first person whose destinationFloor is greater than the given floor number. Uses Array.splice() for removal. Returns null if no match is found.
  • dequeueNextWithDestinationBelow(floor): Same as above but finds the first person whose destinationFloor is less than the given floor number.
Testing Ideas
  • Test dequeueNextWithDestinationAbove: let fq = new FloorQueue(); let p1 = new Person('a', 0, 2); let p2 = new Person('b', 0, 1); fq.enqueue(p1); fq.enqueue(p2); console.assert(fq.dequeueNextWithDestinationAbove(0) === p1 && fq.size() === 1);
  • Test dequeueNextWithDestinationBelow: let fq2 = new FloorQueue(); fq2.enqueue(new Person('c', 2, 0)); console.assert(fq2.dequeueNextWithDestinationBelow(2).destinationFloor === 0);

4. ElevatorQueue Class

File: js/queue.js

Why build this next: ElevatorQueue extends Queue to dequeue people based on their exact destination floor, used in elevators. It depends on Queue.

Next: Define the DIRECTION constants, as they are used in the Elevator class.

A specialized queue for elevators, extending Queue.

Methods:

  • Inherits constructor from Queue.
  • dequeue(floor): Overrides the base dequeue(). Iterates through the queue and removes and returns the first person whose destinationFloor exactly matches the given floor number. Uses Array.splice() for removal. Returns null if no match is found.
Testing Ideas
  • Test dequeue by floor: let eq = new ElevatorQueue(); let p1 = new Person('a', 0, 2); let p2 = new Person('b', 0, 1); eq.enqueue(p1); eq.enqueue(p2); console.assert(eq.dequeue(1) === p2 && eq.size() === 1);

5. DIRECTION Constants

File: js/elevator.js

Why build this next: The DIRECTION constants define UP and DOWN values used in elevator movement logic. Define these before Elevator since it references them.

Next: Build the Floor class, which uses FloorQueue.

A frozen object that holds direction constants used by the Elevator class.

  • DIRECTION.UP: Value 100. Indicates the elevator is moving upward.
  • DIRECTION.DOWN: Value 200. Indicates the elevator is moving downward.
Testing Ideas
  • Test constants: console.assert(DIRECTION.UP === 100 && DIRECTION.DOWN === 200);

6. Floor Class

File: js/elevator.js

Why build this next: The Floor class represents individual floors in the building and uses FloorQueue to manage waiting people. It depends on FloorQueue and Person.

Next: Build the Elevator class, which interacts with floors and uses ElevatorQueue.

Represents a floor in the building, managing people waiting and button presses.

Properties:

  • floorNumber: The integer number of the floor (0-indexed).
  • queue: A FloorQueue instance holding people waiting on this floor.
  • upButtonPressed: Boolean indicating if someone on this floor wants to go up.
  • downButtonPressed: Boolean indicating if someone on this floor wants to go down.

Methods:

  • constructor(floorNumber): Initializes the floor with the given floor number, an empty FloorQueue, and both buttons set to false.
  • getPeopleOnFloor(): Returns the number of people waiting on the floor via queue.size().
  • addPerson(person): Adds a person to the floor's queue.
  • removePerson(person): Stub method (not implemented).
  • pressButtons(): Resets both buttons to false, then iterates through all people in the queue. Sets upButtonPressed = true if any person's destination is above this floor, and downButtonPressed = true if any person's destination is below this floor.
Testing Ideas
  • Test constructor: let f = new Floor(0); console.assert(f.floorNumber === 0 && f.queue.isEmpty() === true);
  • Test addPerson and getPeopleOnFloor: let p = new Person('a', 0, 1); f.addPerson(p); console.assert(f.getPeopleOnFloor() === 1);
  • Test pressButtons: f.pressButtons(); console.assert(f.upButtonPressed === true && f.downButtonPressed === false);

7. Elevator Class

File: js/elevator.js

Why build this next: The Elevator class handles elevator movement, loading/unloading passengers, and destination logic. It depends on ElevatorQueue, Floor (for floors array), and DIRECTION constants.

Next: Build RegularElevator, a subclass of Elevator with fixed capacity.

The base class for elevators, handling movement, loading/unloading passengers, and destination logic.

Properties:

  • capacity: The maximum number of people the elevator can hold.
  • floors: A reference to the building's array of Floor objects (shared, not copied).
  • queue: An ElevatorQueue instance holding the people currently riding in the elevator.
  • currentFloor: The floor the elevator is currently on (starts at 0).
  • destinationFloor: The floor the elevator is heading to (starts at 0).

Methods:

  • constructor(capacity, floors): Initializes the elevator with the given capacity, a reference to the building's floors array, an empty ElevatorQueue, and both currentFloor and destinationFloor set to 0.
  • isFull(): Returns true if queue.size() >= capacity.
  • move(): If currentFloor === destinationFloor, returns false (didn't move). Otherwise, increments or decrements currentFloor by 1 toward destinationFloor and returns true.
  • getNextDestinationFloor(): Determines the next floor the elevator should target.
    • If the elevator is empty: scans all floors (skipping the current floor) and returns the first floor number that has either upButtonPressed or downButtonPressed set to true. If no buttons are pressed, returns the current floor (stay put).
    • If the elevator has passengers: returns the destinationFloor of the person at the front of the queue (queue.peek()).
  • getNextDirection(): Calls getNextDestinationFloor() and returns DIRECTION.UP if the next destination is above the current floor, otherwise DIRECTION.DOWN.
  • unLoadPeople(): Attempts to remove one person from the elevator queue whose destinationFloor matches currentFloor. Returns true if someone was unloaded, false otherwise. Only unloads one person per call.
  • loadPeople(): If the elevator is not full, attempts to load one person from the current floor's queue:
    • If heading UP: calls floors[currentFloor].queue.dequeueNextWithDestinationAbove(currentFloor).
    • If heading DOWN: calls floors[currentFloor].queue.dequeueNextWithDestinationBelow(currentFloor).
    If a person is found, enqueues them into the elevator's queue and returns true. Otherwise returns false. Only loads one person per call.
  • moveAndLoad(): The main per-step logic. Tries to move() the elevator. If the elevator didn't move (it's at its destination):
    1. Tries to unLoadPeople(). If no one was unloaded:
    2. Tries to loadPeople(). If no one was loaded:
    3. Sets destinationFloor = getNextDestinationFloor().
    Note: only one action happens per step (move, unload, or load).
Testing Ideas
  • Test constructor: let floors = [new Floor(0), new Floor(1)]; let e = new Elevator(4, floors); console.assert(e.capacity === 4 && e.currentFloor === 0);
  • Test isFull: console.assert(e.isFull() === false);
  • Test move: e.destinationFloor = 1; console.assert(e.move() === true && e.currentFloor === 1);
  • Test getNextDirection: e.destinationFloor = 2; console.assert(e.getNextDirection() === DIRECTION.UP);

8. RegularElevator Class

File: js/elevator.js

Why build this next: RegularElevator is a simple subclass of Elevator with a fixed capacity. It depends on Elevator.

Next: Build the Building class, which manages collections of floors and elevators.

A subclass of Elevator with a fixed capacity of 4 people.

Properties:

  • Inherits all from Elevator, with capacity hardcoded to 4.

Methods:

  • constructor(floors): Calls super(4, floors) to initialize a RegularElevator with a capacity of 4.
Testing Ideas
  • Test constructor: let floors = [new Floor(0), new Floor(1)]; let re = new RegularElevator(floors); console.assert(re.capacity === 4);

9. Building Class

File: js/elevator.js

Why build this next: The Building class manages the overall structure, creating floors and elevators. It depends on Floor and RegularElevator.

Next: Build the renderer classes, then the Simulation class.

Manages the overall building, including floors and elevators.

Properties:

  • numElevators: Number of elevators in the building.
  • numFloors: Number of floors in the building.
  • elevators: Array of RegularElevator instances, each initialized with a reference to this.floors.
  • floors: Array of Floor instances numbered 0 through numFloors - 1.

Methods:

  • constructor(numElevators, numFloors): Creates the floors array first, then creates the elevators array. Each elevator receives the shared floors array.
  • moveElevatorsAndLoad(): Iterates through all elevators and calls moveAndLoad() on each one.
  • getFloor(floorNumber): Returns the Floor object matching the given floor number using Array.find().
  • getNumberFloors(): Returns this.floors.length.
  • generatePeopleRandomly(numPeople): Creates the specified number of people with:
    • A random single-letter name from "a" through "z".
    • A random origin floor from 0 to numFloors - 1.
    • A random destination floor (guaranteed different from origin via a do-while loop).
    Each person is added to their origin floor via getFloor(originFloor).addPerson(person).
Testing Ideas
  • Test constructor: let b = new Building(1, 3); console.assert(b.numElevators === 1 && b.numFloors === 3 && b.floors.length === 3 && b.elevators.length === 1);
  • Test getFloor: console.assert(b.getFloor(1).floorNumber === 1);
  • Test generatePeopleRandomly: b.generatePeopleRandomly(2); console.assert(b.floors.some(f => f.getPeopleOnFloor() > 0));

10. ElevatorConsole Class (Renderer)

File: js/elevatorconsole.js

Why build this next: ElevatorConsole is a simple text-based renderer that outputs the simulation state to the browser console. It has no dependencies beyond the building object passed to render(). Building this before the canvas renderer lets you verify the simulation logic visually without needing canvas drawing code.

Next: Build the ElevatorCanvas renderer for the graphical display.

A standalone console renderer that prints a text-based view of the building each simulation step.

Properties:

  • timer: An integer counter that increments each time render() is called.

Methods:

  • constructor(): Initializes timer to 0.
  • render(building): Clears the console, increments the timer, then loops through each floor from top to bottom. For each floor, builds a text line showing:
    • The floor number.
    • For each elevator: if it's on this floor, shows the destination floors of its passengers in brackets (e.g., [1 2]); otherwise shows empty brackets [ ].
    • The destination floors of people waiting on this floor in parentheses (e.g., (0 2)).
    Finally, logs the timer value.

Expected Building Interface:

  • building.floors: Array of floor objects, each with .floorNumber and .queue.people (array of objects with .destinationFloor).
  • building.elevators: Array of elevator objects, each with .currentFloor and .queue.people (array of objects with .destinationFloor).
Example Console Output

Floor 1: [0] [ ] ()
Floor 0: [ ] [ ] (2 1)
Timer: 5

11. ElevatorCanvas Class (Renderer)

File: js/elevatorcanvas.js

Why build this next: ElevatorCanvas draws a graphical representation of the simulation onto an HTML canvas. It has no dependencies on the simulation classes other than the building object interface passed to render().

Next: Build the Simulation class to tie everything together.

A standalone canvas renderer that draws a retro-futuristic visualization of the building.

Properties:

  • canvas: Reference to the HTML <canvas> element.
  • ctx: The 2D rendering context from the canvas.
  • timer: An integer counter that increments each render call.
  • domStats (optional setter): An object with references to DOM elements for displaying stats: { timer, waiting, transit }.
  • colors: A color palette object with named color values for the retro-futuristic theme: bgDeep, bgPanel, amber, amberDim, amberGlow, cyan, cyanDim, cyanGlow, red, redGlow, gridLine, textDim, shaftBg.

Methods:

  • constructor(canvas): Stores the canvas reference and gets the 2D context. Initializes timer to 0.
  • render(building): The main draw method. Each call:
    1. Increments the timer.
    2. Calculates layout dimensions: padding, shaft width, floor height, shaft start position.
    3. Updates DOM stat elements (if domStats is set) with total waiting people, total in-transit people, and the timer.
    4. Clears the canvas and draws the background grid.
    5. Draws the building outline.
    6. For each floor (bottom to top): draws floor separator lines, floor number labels, up/down call button indicators (triangles that glow cyan when up is pressed, red when down is pressed), and person figures for everyone waiting.
    7. For each elevator shaft: draws the shaft background, guide rails, floor indicators, and the elevator car. The car border color reflects state (cyan = ascending, red = descending, amber = idle). Inside the car, draws passenger figures and a capacity indicator (e.g., "2/4").
    8. Draws corner bracket decorations.
  • drawPerson(ctx, x, y, destFloor, goingUp, inElevator): Draws a geometric humanoid figure at the given position. Color is cyan if going up, red if going down. Includes a glow effect and a circular badge on the body showing the person's destination floor number. The inElevator flag slightly reduces the figure size.
  • drawCornerBracket(ctx, x, y, size, color, flipX, flipY): Draws an L-shaped corner bracket decoration. Used at the four corners of the building outline for visual flair.

Expected Building Interface:

  • building.floors: Array of floor objects, each with .floorNumber, .upButtonPressed, .downButtonPressed, and .queue.people (array of objects with .destinationFloor).
  • building.elevators: Array of elevator objects, each with .currentFloor, .destinationFloor, .capacity, and .queue.people (array of objects with .destinationFloor).

12. Simulation Class

File: js/elevator.js

Why build this last: The Simulation class ties everything together, managing the simulation loop, timing, and rendering. It depends on Building, ElevatorCanvas, and ElevatorConsole.

Manages the entire simulation, including timing, running/pausing, and rendering.

Properties:

  • numElevators: Number of elevators.
  • numFloors: Number of floors.
  • peoplePerSecond: Number of new people added per simulation step.
  • isRunning: Boolean indicating if the simulation is currently running.
  • intervalId: The ID returned by setInterval, used to pause the simulation.
  • building: A Building instance created with the given elevator and floor counts.
  • renderers: An array of renderer objects. Always includes an ElevatorConsole. If a canvas element is passed to the constructor, also includes an ElevatorCanvas.

Methods:

  • constructor(numElevators, numFloors, peoplePerSecond, canvas): Initializes all properties. Creates the Building. Populates the renderers array with an ElevatorCanvas (if canvas is provided) and an ElevatorConsole.
  • set domStats(stats): A setter that forwards DOM stat references to renderers that support them (specifically the ElevatorCanvas renderer).
  • step(): Advances the simulation by one step:
    1. Calls building.generatePeopleRandomly(peoplePerSecond) to add new people.
    2. Calls pressButtons() on every floor to update button states.
    3. Calls building.moveElevatorsAndLoad() to advance all elevators.
    4. Calls render(building) on each renderer.
  • toggleRun(): If running, calls pause(); otherwise calls run().
  • run(): Starts the simulation loop by calling setInterval(() => this.step(), 1000) (one step per second). Sets isRunning = true.
  • pause(): Stops the simulation loop by calling clearInterval(intervalId). Sets isRunning = false.
Testing Ideas
  • Test constructor: let canvas = document.createElement('canvas'); let sim = new Simulation(1, 3, 1, canvas); console.assert(sim.numElevators === 1 && sim.building instanceof Building);
  • Test toggleRun: sim.toggleRun(); console.assert(sim.isRunning === true); sim.toggleRun(); console.assert(sim.isRunning === false);

HTML Page (elevator.html)

File: elevator.html

The main entry point that sets up the UI and bootstraps the simulation.

Visual Design

The page uses a retro-futuristic "control room" aesthetic with:

  • Fonts: Orbitron (headings/numbers) and JetBrains Mono (body text).
  • Color scheme: Dark background (#0a0a0c), amber accents (#ffb000), cyan highlights (#00ffd0), and red alerts (#ff3366).
  • Effects: Scanline overlay via CSS ::before pseudo-element, grid background via ::after, and pulsing status badge animation.

UI Elements

  • Header: Title "Vertical Transit" with a pulsing "SIMULATION" badge.
  • Control panel: Contains:
    • An "Initialize" button (#runBtn) that toggles between "Pause" and "Resume".
    • Three stat displays: Cycle counter (#timerDisplay), Waiting count (#waitingDisplay), and In Transit count (#transitDisplay).
  • Canvas: A 900x500 pixel canvas (#elevatorCanvas) wrapped in a decorative frame with corner brackets.
  • Legend: Color-coded indicators for Ascending (cyan), Descending (red), and Idle (amber).

JavaScript (inline module script)

The script imports Simulation from ./js/elevator.js and sets up the run button:

  • On first click: creates a new Simulation(2, 3, 1, canvas) — 2 elevators, 3 floors, 1 person per second.
  • Passes DOM stat elements to the simulation via sim.domStats.
  • On subsequent clicks: toggles between running and paused states, updating the button label.

Simulation Flow Summary

Each simulation step (once per second) follows this sequence:

  1. Generate people: peoplePerSecond new people are created with random origins and destinations and placed on their origin floors.
  2. Press buttons: Each floor scans its waiting people and sets upButtonPressed and/or downButtonPressed accordingly.
  3. Move elevators: Each elevator executes moveAndLoad():
    • If the elevator is not at its destination, it moves one floor closer.
    • If the elevator is at its destination, it tries to unload one person. If no one to unload, it tries to load one person. If no one to load, it picks a new destination.
  4. Render: Both renderers (console and canvas) are called with the building state.

How to Recreate the Project

  1. Implement classes in order: Follow the step-by-step class implementation above (steps 1-12), building each class in the specified order to manage dependencies.
  2. HTML setup: Create elevator.html with a canvas element and a button to start/pause the simulation. Include a module script that imports and runs the Simulation class.
  3. Testing: Run the simulation and verify that elevators move correctly, people are loaded/unloaded, and both the canvas and console update properly.

This project uses ES6 modules, so it must be served via a local HTTP server (not opened directly as a file). For visualization, the canvas drawing logic can be adjusted for different layouts or color schemes.
